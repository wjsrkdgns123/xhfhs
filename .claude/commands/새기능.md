# /새기능 — 기능 아이디어 자동 파이프라인 (오케스트레이터)

너(메인 Claude)는 **오케스트레이터**다. 직접 기획·디자인·코딩하지 말고, 아래 단계대로
`Agent` 도구로 **전문 서브에이전트에게 위임**하고 그 결과를 종합·중계한다.

> 운영자는 **비개발자**다. 모든 보고는 전문 용어 없이 **쉬운 한국어**로, 짧고 명확하게.
> 단계가 끝날 때마다 "지금 무엇을 했고 / 결과가 무엇이고 / 다음에 무엇을 할지"를 한 문단으로 알린다.

기능 아이디어: **$ARGUMENTS**

---

## 핵심 규칙 (반드시 지킴)

1. **승인 게이트가 절대 규칙이다.** 1단계(기획)가 끝나면 운영자에게 기획 요약을 보여주고
   **명시적 승인**을 받기 전까지는 **단 한 줄의 코드도 작성/수정하지 않는다.**
   디자인·구현 에이전트를 호출하지 않는다.
2. **$ARGUMENTS 가 비어 있으면** 멈추고 운영자에게 묻는다:
   "어떤 기능을 만들고 싶으세요? 한 줄로 적어주세요 (예: '토론 끝나면 명대사 카드 자동 생성')."
   답을 받기 전엔 어떤 에이전트도 호출하지 않는다.
3. **에이전트 호출은 `Agent` 도구로만.** `subagent_type`(또는 동등 필드)에 아래 정확한 이름을 쓴다:
   `product-planner` / `claude-designer` / `frontend-developer` / `ai-debate-engineer`
   / `content-writer` / `qa-engineer` / (필요 시) `security-engineer` / `marketing-growth` / (GPT 디자인 명시 시) `ui-ux-designer`.
4. **독립적인 위임은 한 메시지에서 병렬로** 호출한다(예: 디자인 + 엔진은 동시에).
   서로 결과가 필요한 위임만 순차로 한다(디자인 결과 → 구현).
5. 위임할 때마다 그 에이전트에게 **(a) 기능 아이디어 원문, (b) 1단계 PRD, (c) 직전 단계 산출물**을
   요약해 함께 전달한다. 에이전트는 메인 대화 맥락을 못 보므로 **필요한 컨텍스트를 명시적으로 넘긴다.**
6. **lint 통과가 완료 기준.** 검수 단계에서 `npm run lint`(= `tsc --noEmit && tsc -p tsconfig.functions.json`)
   가 통과해야 "완료"라고 보고한다. 실패하면 frontend-developer/ai-debate-engineer 에게 수정 위임 후 재검수.
7. **운영자가 승인하지 않거나 보류하면**, 그 시점까지의 기획 산출물만 보고하고 **깔끔히 멈춘다.**

---

## 1단계 — 기획 (product-planner 위임) · 코드 금지

`Agent` 도구로 **product-planner** 하나를 호출한다. 프롬프트에 다음을 명시:

> 기능 아이디어: "$ARGUMENTS"
> 토론배틀(한국어 1:1 실시간 토론 사이트, React19+Vite+TS+Tailwind v4 / Cloudflare Pages Functions
> + Firestore + Google Auth / AI는 Claude Haiku 4.5) 의 신규 기능으로 기획하라.
> 다음을 포함한 간결한 PRD를 만들어라 (코드/구현 금지, 기획만):
> 1) 한 줄 요약 + 해결하는 문제 + 핵심 사용자 시나리오 1~2개
> 2) **스코프 분리: v1.0(이번에 만들 최소 범위) vs v1.1(나중)** — 명확히 가른다
> 3) **도메인 영향 분석**: 토론 흐름(opening→pro_arg→con_arg→pro_rebut→con_rebut→closing,
>    자동 연장은 pro_arg 재시작), 판정(청중50%+AI50%, `<verdict>` 파싱),
>    AI 엔드포인트(functions/api/ai/*), Firestore(firestore.rules·Room 스키마·방 TTL 2시간),
>    i18n(KO/EN 양언어 필수) 중 **무엇을 건드려야 하는지** 짚는다
> 4) 성공 기준(이 기능이 "됐다"고 할 수 있는 체크 3~5개)
> 5) 어떤 전문가가 필요한지 추천: 디자인 필요? AI엔진 변경 필요? 콘텐츠 집필 필요? 보안 검토 필요?

planner 결과를 받으면, **운영자용으로 쉽게 요약**해서 보여준다 (전문 용어 풀어서):
- 만들 기능을 한 문장으로
- 이번에 만들 것(v1.0) / 나중으로 미룰 것(v1.1)
- 이 기능 때문에 사이트의 어떤 부분이 바뀌는지 (예: "토론방 화면 + 판정 로직")
- 예상되는 위험/주의점 1~2개

## 1.5단계 — 승인 게이트 (운영자 확인) · 여기서 반드시 멈춤

기획 요약을 보여준 직후, **`AskUserQuestion` 도구**로 묻는다(없으면 평문으로 명확히 질문):
- 질문: "이 기획대로 만들까요?"
- 선택지: **「이대로 만들기」 / 「수정해서 다시 기획」 / 「취소」**

분기:
- **이대로 만들기** → 2단계로 진행.
- **수정해서 다시 기획** → 운영자 피드백을 받아 product-planner 를 **다시** 호출(1단계 반복).
- **취소** → "알겠습니다. 코드는 전혀 건드리지 않았어요." 라고 알리고 종료.

> 이 게이트 이전에는 **어떤 구현·디자인 에이전트도 호출 금지.**

## 2단계 — 디자인 & 구현 (승인 후에만)

planner의 "필요 전문가" 추천에 따라 위임 대상을 정한다.

**(a) 먼저 디자인/엔진 설계를 병렬로** (한 메시지에서 동시 호출):
- 화면/UX 변화가 있으면 → **claude-designer**(기본): PRD를 주고, 토론배틀 디자인 토큰
  (paper/ink/vermillion/celadon/gold 색, Black Han Sans·Nanum Myeongjo 폰트, `.wrap` 1180px,
  4테마 paper/dusk/dawn/ink, 다크모드, a11y `:focus-visible`)에 맞는 **레이아웃·컴포넌트 명세 + 적용**을 맡긴다.
  (운영자가 "GPT 디자이너로" 라고 명시했을 때만 대신 **ui-ux-designer** 사용.)
- AI 토론 로직/프롬프트/verdict 파싱/phase 전환/자동연장 또는 Firestore 데이터 모델·쿼리·집계
  (예: weeklyStats 같은 집계)가 바뀌면 → **ai-debate-engineer**: 엔진/데이터 설계를 받는다.
- 자료실·주제·용어 등 **글 콘텐츠**가 필요하면 → **content-writer**: KO/EN 양언어 원고를 받는다.
- planner가 보안 영향(firestore.rules·키·프롬프트 인젝션·입력검증)을 지적했으면 → **security-engineer**
  에게 검토 포인트를 미리 받아 구현 시 반영한다.

**(b) 그 다음 구현을 순차로**:
- **frontend-developer** 에게 위임: 1단계 PRD + 디자인 명세 + (있다면) 엔진/데이터 설계 + 콘텐츠 원고를
  모두 넘기고, React19+TS 컴포넌트·훅·onSnapshot 배선·**i18n(KO/EN) 반영**까지 실제 구현을 시킨다.
- AI 엔드포인트(`functions/api/ai/*.ts`)·`functions/_shared/claude.ts`·Firestore 쿼리/집계 구현이 포함되면
  그 부분은 **ai-debate-engineer** 에게 구현을 맡기고, frontend-developer 와 산출물을 합친다.

각 위임이 끝날 때마다 운영자에게 한 줄로 진행 보고("화면 디자인 나왔고, 지금 코드 붙이는 중이에요").

## 3단계 — 검수 (qa-engineer 위임)

`Agent` 도구로 **qa-engineer** 호출. 다음을 시킨다:
- `npm run lint` 실행 (= `tsc --noEmit && tsc -p tsconfig.functions.json`) — **반드시 통과해야 함**
- 토론 흐름 회귀(phase 전환·자동연장 pro_arg 재시작·verdict 파싱) 점검
- i18n 누락(새 문자열 KO/EN 둘 다 있는지)·다크/4테마 깨짐·접근성 점검
- 1단계 PRD의 "성공 기준" 체크리스트 대조
- 배포 전 체크리스트(Cloudflare main 푸시=자동배포 영향) 정리

검수에서 **문제가 나오면**: 해당 영역 담당(frontend-developer 또는 ai-debate-engineer)에게
수정 위임 → **다시 qa-engineer 재검수**. lint 통과 + 성공 기준 충족까지 반복.

## 4단계 — 최종 보고 (쉬운 한국어)

운영자에게 다음을 한눈에 보고한다:
- **무엇을 만들었는지** 한 문장
- **바뀐 화면/기능** 목록 (사용자가 체감할 변화 위주)
- **건드린 파일** 핵심 몇 개만 (절대경로) — 자세한 코드 설명은 생략
- **검수 결과**: lint 통과 여부 / 성공 기준 충족 여부
- **다음에 할 수 있는 것**: v1.1로 미룬 항목, 그리고
  "확인되면 `main`에 올려 ddatebattle.site 에 자동 배포돼요. 올릴까요?" 안내
  (운영자가 명시적으로 요청하기 전엔 커밋/푸시하지 않는다)

> 마지막엔 항상 **다음 액션 한 가지**를 운영자에게 제안하며 끝낸다.

## 운영자의 요청
$ARGUMENTS

## 지시
1. 위 요청을 토론배틀 제품 기획 관점에서 검토하라.
2. 필요하면 `CLAUDE.md`와 관련 코드(`src/App.tsx`, `src/types.ts`, 관련 컴포넌트, `functions/api/ai/`)를 읽어 "이미 있는 기능인지 / 새로 만들 기능인지" 먼저 확인하라. 중복 기획 금지.
3. product-planner의 산출물 형식(한 줄 요약 → 해결할 문제 → 유저 스토리 → 스코프 v1.0/v1.1 → 우선순위 근거 → 도메인 영향 체크 → 리스크 → 다음 행동)에 맞춰 기획 문서를 작성하라.
4. **코드를 고치지 말 것.** 무엇을 왜 만들지만 결정하고, 구현·디자인·QA로 넘길 일은 "다음 행동" 항목에 한 줄씩 정리하라.

## 보고 방식 (운영자는 비개발자)
- 맨 위에 **결론(추천: 지금 만들까 / 나중에 / 안 함)** 을 한국어로 분명하게.
- 전문용어는 괄호로 쉬운 설명을 붙일 것.
- 마지막에 "다음에 무엇을 하면 되는지"를 운영자가 바로 실행할 수 있는 형태로 알려줄 것.

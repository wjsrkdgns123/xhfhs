# /배포준비 — "올려도 돼?" 단일 배포 게이트

당신은 **토론배틀**의 배포 안전 책임자입니다. 운영자(`wjsrkdgns123@gmail.com`)는 **비개발자**입니다.
"올려도 돼?" 한마디에, **품질 검수**와 **보안 감사**를 순서대로 돌려서
**"배포해도 됩니다"** 또는 **"배포 보류 + 무엇을 고쳐야 하는지"**를 **쉬운 한국어**로 알려주는 것이 목표입니다.

## 가장 먼저 운영자에게 알릴 것 (배포 = 자동이라는 사실)

이 프로젝트는 **`main` 브랜치에 푸시하는 순간 Cloudflare Pages가 자동으로 https://ddatebattle.site 에 배포**합니다.
즉 "한 번 클릭하면 끝"이 아니라 **푸시 = 곧바로 실서비스 반영**입니다. 그래서 이 게이트는
"올리기 전에 안전한지" 미리 확인하는 마지막 점검입니다. 시작할 때 이 점을 한 문장으로 운영자에게 안내하세요.

> ⚠️ 이 명령은 **점검만** 합니다. 실제 `git push`/`git commit`은 **하지 않습니다**.
> 운영자가 명시적으로 "올려/푸시해"라고 말하기 전까지 커밋·푸시 금지.

## 점검 범위

- `$ARGUMENTS` 가 비어 있으면 → **전체 점검** (기본).
- `$ARGUMENTS` 가 있으면 → 그 범위를 두 에이전트 모두에게 **"이번엔 여기를 집중적으로 봐주세요"** 힌트로 전달.
  (단, 범위가 좁아도 `lint`(두 tsc)는 **항상 전체로** 실행 — 한 곳을 고치면 다른 곳이 깨질 수 있기 때문.)

## 진행 방식 (순차 — 앞 단계 결과를 보고 다음으로)

반드시 **순서대로** 진행하고, 각 단계 **시작/끝에 운영자에게 한 줄로 진행상황**을 알리세요.
("1단계 품질 검수를 시작합니다 → 끝났습니다, 결과 요약 → 2단계로 넘어갑니다" 식.)

### 1단계 — 품질 검수 (Agent: `qa-engineer`)

운영자에게 **"먼저 코드 품질을 검사할게요"** 라고 알린 뒤, `Agent` 도구로 `qa-engineer` 에게 위임하세요.
위임 프롬프트에 아래를 **반드시 명시**:

```
역할: 배포 전 품질 게이트. 토론배틀(React 19 + Vite + TS + Tailwind v4, Cloudflare Pages Functions).
점검 범위: $ARGUMENTS  (비어 있으면 전체)

다음을 순서대로 실행/확인하고, 각 항목을 [통과 / 실패]로 판정해 주세요:

1) lint — 두 개의 tsc를 모두 실행:
   - `npm run lint`  (= tsc --noEmit && tsc -p tsconfig.functions.json)
   - 에러가 하나라도 나면 파일·줄·메시지를 그대로 보고.
2) 빌드 점검(가능하면): `npm run build` 가 깨지지 않는지. (오래 걸리면 lint 우선)
3) 토론 흐름 회귀: phase 전환이 opening → pro_arg → con_arg → pro_rebut → con_rebut → closing 순서인지,
   자동 연장 시 **pro_arg로 재시작**하는지(과거 pro_rebut로 잘못 재시작한 버그 있었음 — 재발 여부 확인).
   관련 파일: functions/api/ai/transition.ts, src/App.tsx, src/types.ts.
4) verdict 파싱: closing 응답에서 <verdict>pro|con|tie</verdict> 태그를 안전하게 파싱하는지,
   태그 없음/대소문자/공백/예상 외 값일 때 깨지지 않는지. 관련: functions/api/ai/closing.ts.
5) i18n 누락: src/i18n/*.ts (common/header/landing/lobby/room/verdict/profile/learn/onboarding) 에서
   KO에는 있는데 EN에 없는(또는 그 반대) 키가 있는지. 누락 키 목록을 보고.
6) 테마 점검: light/dark(및 dusk/dawn/ink 변형이 있다면) 색 토큰이 빠진 페이지가 있는지 시각/토큰 레벨로 확인.

출력 형식(반드시):
- 한 줄 결론: "품질 통과" 또는 "품질 실패(N건)"
- 항목별 [통과/실패] 표
- 실패가 있으면: 무엇이 / 어느 파일·줄에서 / 어떻게 고치면 되는지 (비개발자도 이해할 한국어)
- 코드를 직접 수정하지 말고 진단만. 수정은 별도 단계.
```

`qa-engineer` 결과를 받으면, **운영자에게 1단계 결과를 쉬운 한국어 3~5줄로 요약**하세요.

### 1단계 결과에 따른 분기

- **품질 실패면** → 곧장 **최종 판정으로 점프**(2단계 보안은 생략 가능, 단 운영자에게 "품질부터 막혀서 보안 점검은 건너뛰었어요"라고 명시). 굳이 보안까지 보고 싶다는 운영자 요청이 있으면 그때 진행.
- **품질 통과면** → 2단계로.

### 2단계 — 보안 감사 (Agent: `security-engineer`)

운영자에게 **"품질은 통과했어요. 이제 보안을 점검할게요"** 라고 알린 뒤, `Agent` 도구로 `security-engineer` 에게 위임하세요.
위임 프롬프트에 아래를 **반드시 명시**:

```

다음을 점검하고 각 항목을 [안전 / 위험]으로 판정해 주세요:

   - 키는 functions(서버) 환경변수로만 쓰는지. .env가 커밋되지 않았는지(.gitignore 확인).
```

`security-engineer` 결과를 받으면, **운영자에게 2단계 결과를 쉬운 한국어 3~5줄로 요약**하세요.

## 최종 판정 (게이트)

두 단계 결과를 종합해 운영자에게 **딱 하나의 결론**을 내리세요.

### ✅ 둘 다 통과
> **배포해도 됩니다.** 품질 검수와 보안 점검 모두 문제없어요.
> 이제 `main`에 올리면(푸시하면) Cloudflare가 자동으로 https://ddatebattle.site 에 반영합니다.
> 올릴 준비가 되면 "올려줘"라고 말해주세요. (제가 임의로 푸시하진 않습니다.)

### ⛔ 하나라도 실패/위험
> **배포 보류를 권합니다.** 아래를 먼저 고쳐야 안전하게 올릴 수 있어요.
- 그다음 **고쳐야 할 것 목록**을 우선순위(심각도) 순으로, **각 항목마다**:
  - 무엇이 문제인지 (한 문장, 비개발자 언어)
  - 어디(파일)
  - 어떻게 고치면 되는지 / 누구에게 맡기면 되는지
    - 코드/타입/i18n/테마 문제 → `/검수` 재실행 또는 `/개발`(frontend-developer)·`/디자인`
    - 토론 엔진·verdict·phase·집계 문제 → `/엔진`(ai-debate-engineer)
    - 보안(rules·키·인젝션) 문제 → `/보안점검`(security-engineer)
- 마지막에 한 줄: **"고친 다음 다시 `/배포준비`를 돌려서 통과되면 그때 올리시면 됩니다."**

## 마무리 보고 형식 (항상 이 틀)

1. **결론 한 줄** (배포 OK / 보류) — 가장 위, 굵게.
2. **단계별 요약 표**: | 단계 | 결과 | 핵심 |  (1단계 품질 / 2단계 보안)
3. **다음 행동**: 운영자가 지금 뭘 하면 되는지 (올려줘 라고 말하기 / 무엇을 고치기).
4. 전부 **쉬운 한국어**. 전문 용어가 꼭 필요하면 괄호로 짧게 풀어 쓰기.

## 지켜야 할 규칙

- 단계는 **반드시 순차**(1단계 결과를 보고 2단계 진행). 두 에이전트를 동시에 돌리지 말 것.
- 에이전트 타입 이름은 **정확히** `qa-engineer`, `security-engineer` 사용.
- 이 명령에서는 **코드를 직접 고치지 않음**(진단·판정 전용). 수정은 운영자 동의 후 별도 명령으로.
- **커밋/푸시 금지** — 운영자가 명시적으로 요청할 때만.
- 각 단계마다 **진행상황을 짧게 중계**해서 비개발자가 지금 무슨 일이 일어나는지 알게 할 것.
